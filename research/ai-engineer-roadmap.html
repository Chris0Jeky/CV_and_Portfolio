The Role — what "AI Engineer" actually means in 2025–26

Before chasing skills, understand the job. The title crystallized over the last two years into something distinct from ML Engineer and Data Scientist.

The one-sentence definition Verified

An AI Engineer deploys and integrates pretrained LLMs into production products — prompt engineering, RAG, agents, tool-use, LLM APIs — rather than training models from scratch. It is fundamentally a software-engineering discipline applied to LLM-based systems.

This is corroborated unanimously across a recruiter salary guide, a 2026 roadmap, and academic/industry sources (U. Manchester, TechTarget, Nucamp). The nuance: AI Engineers do fine-tune (LoRA etc.) — but that adapts a pretrained model, it doesn't train one from zero.

Why this matters for youYou already are a software engineer. The shortest path is to bolt LLM-system skills onto that foundation — you do not need a deep-learning PhD or to learn to train transformers. That's the whole reason this role is accessible to web/backend devs.
What 2026 hiring managers actually screen for Verified

Hiring in 2026 screens for production signals over notebook demos. The specific signals:

  • Failure handling — what happens when the LLM returns garbage, times out, or hallucinates a tool call?
  • System integration — does it talk to real APIs, databases, queues?
  • Eval literacy — described as "the single biggest signal." Can you prove your system works and catch regressions?
  • Shipping working software — deployed, callable, observable.
  • Data structuring — turning messy inputs into reliable structured outputs.
Rejection pattern"Notebook-only experience presented as production is a rejection pattern." A Jupyter notebook with a LangChain demo reads as a red flag if you frame it as production work. Deploy it behind an API, add evals, add a cost dashboard — or don't claim it's production.

The 7 Core Competency Domains

The heart of the brief. Each domain: what it is, the skills to build, the tools to learn (with verified status), and how to prove it in a portfolio. Expand the ones you want to go deep on.

D1 · Production AI Agents & Agentic Workflows Core

What it is: Giving an LLM tools (web search, retrieval, calculators, code execution, your internal APIs) and letting it plan and act in multiple steps to accomplish a goal — reliably, with guardrails, error handling, and the ability to resume/replay.

Skills to build

  • Tool/function calling — schemas, argument validation, handling malformed tool calls
  • Orchestration patterns — single-agent loops, multi-agent handoffs, supervisor/worker, graphs/state machines
  • State & memory — short-term (conversation) vs long-term (persistent), checkpointing, replay
  • Control & safety — max-step limits, human-in-the-loop approval gates, timeouts, retries, fallbacks
  • The hard part: reliability. Agents fail in compounding ways; production work is mostly about constraining and recovering from failure.
Frameworks — what to learn & what to avoid Verified
  • LangGraph Verified — the production standard. Graph-based: nodes = operations, edges = transitions. Gives explicit typed state, auditability, and built-in checkpointing with time-travel replay (confirmed in official LangChain docs). Reached v1.0 GA Oct 2025. Learn this first.
  • CrewAI Consensus — role-based multi-agent; popular for rapid prototyping. (Note: the widely-repeated "CrewAI uses 18% more tokens than LangGraph" stat was refuted — ignore it.)
  • PydanticAI Consensus — type-safe, Pythonic, great DX for structured agents. Strong rising pick.
  • OpenAI Agents SDK Added — lightweight, handoffs-based, good if you're OpenAI-centric. (Specific per-framework feature claims about it were refuted in research — verify against current docs before quoting specifics.)
  • AutoGen → AVOID for new projects Avoid — Microsoft moved it to maintenance mode and superseded it with the Microsoft Agent Framework (public preview Oct 2025, GA ~April 2026). Don't start a 2026 portfolio project on AutoGen.
Anthropic-native optionSince you'll be quizzed on Anthropic: the Claude Agent SDK and native MCP (Model Context Protocol) tool integration are worth a project. MCP is the emerging standard for connecting agents to tools/data — high signal in 2026.
How to prove it (portfolio)

Build a Tool-Calling Agent that uses ≥3 real tools, handles a tool failure gracefully, enforces a step limit, and exposes a trace of its reasoning. Bonus: a human approval gate before any "write" action. See Project 3.

D2 · RAG & Retrieval Engineering Core

What it is: Connecting an LLM to your own data so it answers from facts, not just its training. The ACQ spec literally says "own and improve RAG pipelines across multiple Pinecone namespaces" — RAG is the load-bearing skill for that role.

The RAG pipeline, stage by stage

  1. Ingest & chunk — split documents into retrievable pieces (fixed-size, recursive, semantic, hierarchical, or "contextual" chunking).
  2. Embed — turn chunks into vectors with an embedding model.
  3. Index & store — put vectors in a vector DB (namespaces, metadata filters, index params).
  4. Retrieve — dense (vector) search, sparse (BM25/keyword), or hybrid (both); then filter by metadata.
  5. Rerank — a cross-encoder reorders the top-k for relevance before it hits the prompt.
  6. Generate — stuff retrieved context into the prompt; cite sources; handle "I don't know."
The verified RAG tool stack Verified
LayerLearn firstAlso valid
Vector DBQdrant, then Pinecone · Chroma for local devFAISS, Weaviate, Milvus, pgvector
Embeddingstext-embedding-3-large (OpenAI) · BGE-M3 (open)Voyage, Cohere embed, Nomic, E5
RerankingCohere Rerank (API) · BGE Reranker (self-host)Jina reranker, cross-encoders

Embedding picks are grounded in the MTEB leaderboard (text-embedding-3-large ≈64.6, BGE-M3 ≈63.0). Cohere Rerank + BGE-reranker-v2-m3 are the standard API/self-host pair. "Qdrant first" is an opinionated-but-mainstream pick — Pinecone is what the ACQ job uses, so learn both.

Interview goldThe ACQ role names Pinecone namespaces. Build at least one project on Pinecone with multiple namespaces (e.g. per-tenant or per-document-type isolation) so you can speak to it directly.
Advanced retrieval techniques to learn Caveat

These techniques are real and worth learning — but be careful: the specific "X% recall improvement" numbers floating around blogs were refuted in this research. Learn the techniques; measure the gains yourself with evals.

  • Hybrid search — combine dense + sparse (BM25). Generally improves recall, especially for keyword/jargon-heavy queries. Magnitude is data-dependent — measure it.
  • Reranking — cross-encoder over top-k. Usually the highest-ROI single addition to a naive RAG.
  • Chunking strategy — fixed vs recursive vs semantic vs hierarchical. Matters, but the exact F1 deltas in blog posts were unverified/fabricated-seeming. Test on your corpus.
  • Contextual retrieval Added — Anthropic's technique: prepend chunk-level context before embedding. Worth reading Anthropic's own write-up.
  • Query transformation — HyDE, multi-query, query decomposition for complex questions.
  • Metadata filtering & namespaces — scope retrieval to the right subset before vector search.
How to prove it (portfolio)

A Document Q&A RAG that ingests real docs, retrieves with citations, says "I don't know" when it should, and — crucially — has an eval harness proving retrieval quality. See Project 1.

D3 · LLM Evaluation Frameworks Core Highest hiring signal

What it is: Proving — with data, not vibes — that your LLM system works, and catching it when a prompt/model/retrieval change makes it worse. Research flagged eval literacy as "the single biggest signal" hiring managers look for. This is your highest-leverage skill to differentiate.

What a real eval framework contains

  • Golden dataset — a curated set of inputs with known-good outputs (start ~100 examples). Your ground truth.
  • Automated quality scoring — LLM-as-judge, semantic similarity, rule checks, structured-output validation.
  • Retrieval metrics — recall@k, precision@k, MRR, nDCG, context relevance, faithfulness/groundedness.
  • Latency & cost benchmarks — p50/p95 latency, tokens-per-request, $/request, tracked over time.
  • Regression detection — run evals in CI; gate merges; A/B two prompts/models and compare on the golden set.
Eval tooling Verified
  • RAGAS Verified — the standard for RAG-specific metrics (faithfulness, answer relevancy, context precision/recall). Learn this.
  • DeepEval Verified — pytest-style LLM unit testing; great for CI regression gates.
  • LangSmith Consensus — eval + tracing + datasets, tight LangChain/LangGraph integration.
  • Braintrust Added — eval platform popular at well-funded startups; named as a valid alternative (not independently evaluated here).
  • Arize Phoenix Added — open-source observability + eval; OpenTelemetry-based. Strong for tracing+eval combined.
  • TruLens Added — open-source eval/feedback functions; another valid option.
Minimum viable eval stackRAGAS for RAG metrics + DeepEval for CI assertions + a golden dataset in version control + a dashboard. That combination alone puts you ahead of most candidates.
How to prove it (portfolio)

An LLM Evaluation Pipeline: golden dataset → automated scoring → metrics dashboard → CI gate that fails the build on regression. This is Project 4 and arguably the single most impressive thing in your portfolio.

D4 · Model Routing & Cost / Unit Economics Core

What it is: Spending less per request without losing quality — by sending easy queries to cheap models and hard ones to expensive models, plus caching, token management, and architectural tricks. The ACQ spec: "optimize model routing and tiering to improve unit economics." Production teams care about this a lot.

RouteLLM — the canonical routing technique Verified

RouteLLM (open-source, by LMSYS) is a framework for serving and evaluating LLM routers. Key facts (confirmed against the GitHub repo, LMSYS blog, and the ICLR 2025 arXiv paper):

  • Drop-in replacement for OpenAI's client — minimal code change.
  • Routes simple queries to a cheaper weak_model, complex ones to a stronger strong_model, via a cost-threshold parameter.
  • Advertised: up to 85% cost reduction on MT-Bench while keeping ~95% of GPT-4 performance. (The "85%" is the project's own benchmark-specific number — quote it as "up to … on MT-Bench," not as a universal guarantee.)

This is the strongest-evidenced finding in the whole research pass. Building a routing demo with RouteLLM is a high-signal portfolio move.

The full cost-optimization toolkit Added
  • Model tiering / routing — RouteLLM, or your own classifier (cheap model decides difficulty).
  • Prompt caching — Anthropic & OpenAI both offer it; huge savings on repeated system prompts/context. (Highly relevant if you're building on Claude.)
  • Semantic caching — cache by meaning, not exact string (GPTCache-style).
  • Token management — trim context, compress history, summarize, cap output tokens.
  • Batching — batch APIs for non-realtime workloads (often ~50% cheaper).
  • Right-sizing — use the smallest model that passes your evals (this is why evals + cost are joined at the hip).
The senior framingCost work is where engineering meets business. "I reduced our LLM spend 60% while holding quality on the golden set" is the exact sentence that gets a senior offer. It requires evals (D3) + observability (D5) to prove. These three domains compound.
D5 · Observability & Instrumentation Core

What it is: Seeing inside your AI layer in production — every request's tokens, cost, latency, prompt, retrieval, and quality — so you can debug, detect anomalies, and report. The ACQ spec: "instrument the AI layer for observability — cost-per-request, token usage, quality signals, anomaly detection."

What to instrument

  • Tracing — full request trace: prompt → retrieval → tool calls → completion (spans, like distributed tracing for LLMs).
  • Cost & tokens — $/request, input/output tokens, per-feature and per-user rollups.
  • Latency — p50/p95/p99, time-to-first-token for streaming.
  • Quality signals — online evals, user feedback (thumbs), error/refusal rates.
  • Anomaly detection — alert on cost spikes, latency regressions, quality drops.
Observability tooling Verified
  • LangSmith Verified — tracing + evals + datasets, deep LangChain/LangGraph integration.
  • Langfuse Verified — open-source LLM observability; tracing, metrics, prompt management. Self-hostable.
  • Helicone Verified — proxy-based observability; drop-in, cost & usage analytics.
  • Arize Phoenix Added — OpenTelemetry-native tracing + eval, open-source.
  • LiteLLM callbacks Verified — emits to Lunary, MLflow, Langfuse (see D6).
Standards angleLearn OpenTelemetry for LLMs (OpenLLMetry / OTel GenAI semantic conventions). Vendor-neutral instrumentation is a senior signal and future-proofs your work.
D6 · Multi-Provider LLM Integration Core

What it is: Working fluidly across OpenAI, Anthropic, Google, open models, etc. — swapping providers without rewrites, understanding each one's prompt quirks, token economics, and failure modes. The ACQ spec asks for exactly this.

LiteLLM — the canonical integration layer Verified

LiteLLM (open-source, BerriAI, 47k+ stars) — confirmed against the repo and official docs:

  • A single unified OpenAI-format interface to 100+ providers (OpenAI, Anthropic, Azure, Bedrock, Vertex, Cohere, HuggingFace…). Swap providers without code rewrites.
  • Its proxy/gateway adds production features: spend/cost tracking, load balancing, routing & fallbacks, caching, virtual keys, guardrails, admin dashboard.
  • Observability callbacks to Lunary, MLflow, Langfuse — so it doubles as your instrumentation hook (ties into D5).

This single tool covers multi-provider and observability and cost — extremely high ROI to learn.

What to actually understand (the senior part) Added
  • Prompt-behavior differences — the same prompt behaves differently on Claude vs GPT vs Gemini (system-prompt handling, JSON/tool reliability, refusal styles).
  • Token economics — per-provider input/output pricing, context windows, prompt-caching discounts.
  • Failure modes — rate limits, timeouts, content filters, malformed tool calls, truncation — and how to retry/fallback across providers.
  • Structured output — JSON mode, function/tool calling, and constrained decoding differ by provider.
D7 · The Production Software Foundation (TS + Python, APIs, deploy) Core

What it is: The "7+ years shipping production software" base — distributed backends, APIs, deployment pipelines, monitoring. This is table stakes; the AI domains sit on top of it. The ACQ stack is explicitly TypeScript + Python.

Python side (the AI-native ecosystem)

  • Async Python — asyncio, async/await, concurrency for I/O-bound LLM calls.
  • Type hints + Pydantic — structured I/O, validation; the backbone of reliable LLM apps.
  • FastAPI — the default for serving AI features behind an API.
  • Provider SDKs: openai, anthropic.
TypeScript side of the AI stack Added · research gap

The research pass was Python-heavy and didn't surface verified TS tooling — but the ACQ job uses both, so here's the well-known TS/JS AI stack (general knowledge, verify current versions yourself):

  • Vercel AI SDK — the dominant TS toolkit for streaming, tool-calling, and provider-agnostic LLM calls in JS/TS apps.
  • LangChain.js & LangGraph.js — JS ports of the Python frameworks.
  • LlamaIndex.TS — RAG framework for TS.
  • Mastra — TS-native agent framework, rising fast.
  • Genkit (Google) — TS-first GenAI framework.
Your edgeYou're already a strong JavaScript developer (this very portfolio is hand-rolled vanilla JS). The Vercel AI SDK is your fastest on-ramp — you can build a real, streaming, tool-calling AI feature in the language you already know, today.
Deployment & ops Added
  • Containerization — Docker; basic Kubernetes literacy.
  • Serverless / edge — Vercel, Cloudflare Workers, AWS Lambda for AI endpoints.
  • CI/CD — GitHub Actions running your eval suite as a gate.
  • Streaming — SSE/WebSockets for token streaming to the UI.
  • Secrets & rate limiting — API key management, per-user quotas, queue/backpressure.

Gaps, Myths & Honest Caveats

The integrity section. What the research couldn't confirm, and the specific claims it caught and killed — so you never repeat a fabricated number.

⚠ Refuted claims — do NOT cite these Avoid

These appeared in blogs and were killed by adversarial verification (0–3 or 1–2 votes). The underlying techniques are real and worth learning — but the numbers are unsupported. Don't quote them in interviews or your README:

  • Killed 0-3 "Vector DB p50 latency: Qdrant 6ms / Pinecone 8ms / Weaviate 12ms / Chroma 18ms." Don't use for DB selection — benchmark yourself.
  • Killed 0-3 "Semantic chunking improves F1 by 36%; hierarchical gives 3–5×." Chunking matters; these exact deltas were fabricated-seeming.
  • Killed 1-2 "Hybrid search lifts recall@10 from 78% → 91%." Hybrid helps; the specific numbers are unverified.
  • Killed 0-3 "CrewAI uses ~18% more tokens than LangGraph." Ignore this comparison.
  • Killed 0-3 Specific OpenAI/Anthropic Agents-SDK feature claims (handoffs, "model-locked", "most robust error handling"). Verify against current docs.
  • Killed 0-3 "RAG is THE #1 most in-demand AI skill for 2026." RAG is core but not provably ranked first.

Lesson: the AI-content ecosystem is full of confident, unsourced benchmark numbers. Treat every "X% improvement" as a hypothesis to test on your own data — which is itself the eval mindset that gets you hired.

Open questions the research didn't resolve Caveat
  • TypeScript AI stack — surviving evidence was Python-centric. The TS tools listed (Vercel AI SDK, LangChain.js, LlamaIndex.TS, Mastra, Genkit) are general knowledge, not verified here — confirm current state yourself.
  • Empirical RAG numbers — real, primary benchmarks for chunking and hybrid-vs-dense retrieval (try official Anthropic contextual-retrieval, Qdrant, Pinecone studies).
  • Vector DB / eval-tool head-to-heads — FAISS, Weaviate, Milvus, Braintrust, Arize Phoenix were named as valid but not directly compared.
  • Eval implementation depth — exact metrics (recall@k, MRR, nDCG) and CI-gating patterns are named at a high level; go to RAGAS/DeepEval docs for the details.
  • Comp precision — firm up with Levels.fyi raw filings / H1B data; figures are mid-2026 and shift quarterly.

Tie your technical story to these: "I instrument and evaluate everything because I'd rather the data deliver candid feedback than ship on vibes" hits two of their three values at once.
